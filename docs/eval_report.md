## Non-Determinism Handling

The Sentinel eval system addresses non-determinism in two ways:

1. **pass^k metric:** Rather than evaluating a single run, we run k=8 independent trials and measure whether ALL succeed. This surfaces inconsistency that pass@1 hides.

2. **Judge calibration:** Every LLM judge call is run twice with swapped response ordering. If the verdict flips, it is marked `uncertain` and flagged for human review. This detects and controls for position bias in the LLM judge.

## Test Protocol

1. `make seed` — seeds 100 synthetic JSM incidents across 10 root-cause categories (10 incidents per category, with varied phrasing)
2. The `incident-rca-agent` processes each incident via webhook
3. The flight recorder captures every agent run
4. The eval engine evaluates each captured trace (k=8 trials per incident)
5. Results are aggregated and written to this file

---

## Limitations

- Test dataset is synthetic, not from a real enterprise environment
- The LLM judge (Claude) is not independently validated against human annotations
- Drift detection baseline is derived from the same synthetic dataset used for evaluation
- pass^k with k=8 may not reflect true production reliability at higher k values
